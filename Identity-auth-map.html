Era 01

On-Prem Foundation

Inside the corporate walls. A directory stores everyone; protocols authenticate against it and query it.

Era 02

Cloud / Web

Identity on the internet. The authZ/authN split becomes razor-sharp: OAuth grants access, OIDC proves identity, SAML does enterprise SSO.

NOT authentication. Answers "what can this app do?"

The Currency

Tokens & JWTs

The cloud-era version of the Kerberos ticket. After login you carry a token instead of re-sending your password. Three kinds — and the split maps onto authN vs authZ.

ID Token
authN · OIDC
Says who you are. For the app. Usually a JWT.
Access Token
authZ · OAuth
Says what you can access. For the API / resource server.
Refresh Token
long-lived
Quietly fetches a new access token — no re-login needed.

Stateful

Session

You hold a meaningless session ID; the real data lives on the server. The server must look you up every request. Like a coat-check ticket.

Stateless

Token / JWT

The signed data travels inside the token. The server just verifies the seal — no lookup, no stored state. Like a sealed VIP badge. Scales across many servers.

The Dance

OAuth / OIDC Login Flow

"Sign in with Google" to a photo app. Players: you (resource owner) · the app (client) · Google login (authorization server) · Google Photos API (resource server).

You click "Sign in with Google" on the app.

App redirects you to Google with its client_id, requested scopes, and a PKCE code_challenge.

You authenticate directly with Google. ⭐ The app never sees your password — only Google does.

Google asks you to consent: "This app wants to view your photos. Allow?"

Google redirects you back to the app with a one-time authorization code (useless on its own).

App's backend swaps the code (over a secure back-channel, with PKCE) for the real tokens.

App now holds an ID token (who you are — OIDC) + an access token (call the API — OAuth).

Spans All Eras

The Security Layer — MFA

Extra proof on top of any login. MFA is the strategy; the rest are mechanisms that plug into it. True MFA combines different factor types: something you know · have · are.

MFA
Security ControlEverywhere
The strategy: require more than one type of proof.
Umbrella over everything below.
TOTP
Mechanismunder MFA
Time-based One-Time Password — 6-digit code, refreshes every 30s.
Google / MS Authenticator. Factor: have.
Push Auth
Mechanismunder MFA
Approve a login prompt on your phone — one tap.
User-friendly. Factor: have.
Era 03

Passwordless

Don't add a factor — remove the password. Public-key crypto: your device keeps the private key, the site keeps only a useless public key.

WebAuthn / FIDO2
Standard · authNPasswordless
Site sends a challenge; your device signs it with a private key that never leaves the device.
Phishing-resistant — credential bound to the real domain.
Passkey
CredentialSyncable
A FIDO2 credential that syncs across your devices (iCloud / Google).
One tap = already MFA: device (have) + biometric (are).
Authenticators
where keys live
Platform: Face ID, Windows Hello. Roaming: YubiKey.
Private key sits in secure hardware.
⮑

Three wins: phishing-resistant, nothing secret stored on the server, and no reusable secret crosses the wire.
